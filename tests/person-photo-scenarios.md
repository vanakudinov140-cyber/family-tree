# Person photo — test scenarios

Ручные проверки безопасной загрузки / обрезки / замены / удаления фотографий.
Playwright не обязателен, если Chromium недоступен.

Перед тестами: создан private bucket `person-photos`, применена миграция `005_person_photos.sql` (см. `PHOTO_STORAGE_SETUP.md`).

| # | Сценарий | Ожидание |
|---|----------|----------|
| 1 | Admin загружает JPEG | Фото сохраняется, портрет в профиле |
| 2 | Admin загружает PNG | То же |
| 3 | Обрезка 1:1 | Cropper квадратный; итог круглый портрет |
| 4 | В Storage не оригинал | Файл WebP (или JPEG fallback), размер заметно меньше исходника |
| 5 | Фото в профиле | Большой круглый портрет, без скачка размеров |
| 6 | Фото в heritage node | `object-fit: cover`, размер карточки не меняется |
| 7 | Realtime во 2-й вкладке | UPDATE `photo_path` обновляет UI без fitView / сброса focus |
| 8 | Замена не удаляет старое до RPC | При успехе: сначала новый файл + RPC, потом удаление старого |
| 9 | Ошибка RPC | Новый файл удаляется; старое фото и `photo_path` без изменений |
| 10 | Успешная замена | Старый файл удалён (или warning, если Storage cleanup fail) |
| 11 | Удаление фото | Монограмма; `photo_path = null` |
| 12 | Удаление человека | `deleted_photo_path` → best-effort Storage cleanup; человек не «возвращается» |
| 13 | Relative видит фото, без кнопок | UI управления скрыт |
| 14 | Relative не вызывает RPC | `set_person_photo` / `clear_person_photo` → отказ |
| 15 | Relative не upload Storage | INSERT в `person-photos` отклонён |
| 16 | Гость | Только монограмма; signed URL не запрашивается |
| 17 | SVG | Отклонён на клиенте |
| 18 | Файл > 12 MB (исходник) | «Файл слишком большой» |
| 19 | Картинка < 300×300 | «Изображение слишком маленькое» |
| 20 | Двойное «Сохранить» | Одна операция загрузки |
| 21 | Signed URL не в БД | В `people` только `photo_path`, не URL |
| 22 | Истечение кэша | Signed URL обновляется без перезагрузки страницы |
| 23 | Смена zoom | Layout карточек не пересчитывается из‑за фото |
| 24 | Ошибка загрузки img | Fallback на монограмму |
| 25 | Diagram mode | Рабочий; допустима маленькая круглая фотография |
| 26 | URL / focus / selected / viewMode | Не сбрасываются при photo UPDATE |
| 27 | Длинное ФИО + фото | Нет collision карточек из‑за фото |
| 28 | Mobile bottom sheet + photo dialog | Оба диалога открываются/закрываются корректно |

## Дополнительно

- После замены путь в Storage содержит **новый** `file_uuid`.
- Backup JSON может содержать `photo_path`, но не изображения и не signed URL.
- Orphan-файлы ищутся вручную (см. `PHOTO_STORAGE_SETUP.md`).
