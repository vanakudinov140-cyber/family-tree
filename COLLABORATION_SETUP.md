# Совместное наполнение семейного дерева — настройка

Ручная настройка ролей, предложений и администрирования.
**Не выполняйте автоматически** — только в Supabase SQL Editor и через UI после деплоя.

---

## 1. Выполнить миграцию 006

1. Убедитесь, что применены миграции **001–005**.
2. Откройте `supabase/migrations/006_collaboration_roles_and_proposals.sql`.
3. Скопируйте весь SQL в **SQL Editor** и выполните.
4. Проверьте отсутствие ошибок.

Миграция добавляет роль `editor`, таблицы предложений и журнала, RPC и обновляет политики Storage/RPC.

---

## 2. Проверить текущего admin

```sql
select id, email, role from public.profiles where role = 'admin';
```

Или используйте `supabase/promote_first_admin.sql` для первого назначения.

---

## 3. Зарегистрировать второго пользователя

Через существующую форму регистрации в приложении. Новый пользователь получает роль **relative** автоматически.

---

## 4. Назначить editor через UI

1. Войдите как **admin**.
2. Меню пользователя → **Администрирование** → вкладка **Пользователи**.
3. Выберите пользователя → **Изменить роль** → **Редактор**.

---

## 5. Назначить admin

Тот же экран → роль **Администратор** → введите подтверждение `НАЗНАЧИТЬ АДМИНИСТРАТОРОМ`.

---

## 6. Почему нельзя назначить admin до регистрации

Роль хранится в `public.profiles`, которая создаётся только после `auth.users`. Нельзя назначить роль email, для которого нет аккаунта.

---

## 7. Relative отправляет предложение

1. Войти как relative.
2. Открыть профиль человека → **Предложить изменение**.
3. Выбрать тип (сведения, родственник, связь, биография, фото).
4. Заполнить форму и отправить.

Дерево **не меняется** до решения admin.

---

## 8. Admin принимает предложение

1. **Администрирование** → **Предложения**.
2. Открыть предложение → сравнить «Было / Предлагается».
3. **Принять** / **Уточнить** / **Отклонить**.

---

## 9. «Нужно уточнение»

Admin указывает комментарий → статус `needs_info`. Автор видит его в **Мои предложения** и может ответить.

---

## 10. Добавить родственника через предложение

Тип **Новый родственник**: указываются данные человека и связь с существующим anchor. При approve создаются человек и связь атомарно.

---

## 11. Предложить фотографию

Фото обрезается локально, загружается в `proposals/<user_uuid>/...`, создаётся предложение `photo_replace`. При approve admin копирует в официальный путь `people/<person_uuid>/...`.

---

## 12. Editor редактирует напрямую

Editor видит кнопки **Добавить / Редактировать / Фото** без предложений. Изменения применяются сразу через существующие RPC.

---

## 13. Что editor не может

- удалять людей;
- импорт семьи;
- управлять ролями;
- проверять чужие предложения;
- читать audit log.

---

## 14. Защита последнего admin

`admin_set_user_role` запрещает:
- понизить последнего admin;
- понизить самого себя;
- оставить систему без admin.

---

## 15. Audit log

**Администрирование** → **Журнал**. Только admin. Клиент не может писать в журнал напрямую.

---

## 16. Откат миграции без удаления семейных данных

```sql
-- Удалить RPC и таблицы предложений/журнала (данные people/relationships не трогаются)
drop function if exists public.submit_family_change_proposal cascade;
-- ... остальные новые RPC
drop table if exists public.family_change_proposal_messages;
drop table if exists public.family_change_proposals;
drop table if exists public.family_audit_log;

alter table public.profiles drop constraint if exists profiles_role_check;
alter table public.profiles add constraint profiles_role_check
  check (role in ('relative', 'admin'));

-- Вернуть RPC create_relative/update_person/set_person_photo с is_admin() из миграций 004/005
```

Файлы предложений в Storage удаляются отдельно в Dashboard.
